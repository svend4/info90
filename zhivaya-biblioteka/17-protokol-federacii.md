# 17. Протокол федерации Живой Библиотеки

> Спецификация обмена между инстансами Библиотеки: как два независимых
> сервера узнают друг о друге, синхронизируют канон, не смешивают власть
> и переживают смерть одного из них. Развивает документ 10 §10.5
> и реализован в прототипе (`federate.py`, `export.py`, `deadman.py`).
> Статус: протокол уровня 0 (статические документы), уровень 1 — будущий
> живой сервер; форматы уровня 0 не изменятся.

## 17.1. Принципы

1. **Канон федеративен, власть — нет.** Карточки и знания реплицируются
   свободно; роли (хранители, садовники) существуют только внутри своего
   инстанса. Чужой хранитель на вашем инстансе — просто читатель.
2. **Идентичность = роль + инстанс.** `marina_k@zh-lib.local` и
   `marina_k@lib2.example` — разные субъекты. Коллизии ников не существует,
   потому что ник без инстанса бессмысленен.
3. **Реплика всегда помечена происхождением.** Карточка, полученная
   с другого инстанса, показывает: источник, версию, дату проверки
   и хранителя *там*. Доверие адресное, а не абстрактное.
4. **Федерация не обязательна.** Автономный инстанс — полноценная
   Библиотека; федерация — слой обмена, а не условие работы.

## 17.2. Идентичность и документы

Инстанс публикует по известным путям статические документы
(ActivityStreams 2.0, JSON):

```
/federation/actors.json            # роли инстанса (Person)
/federation/<card_id>.json         # карточка как Article
/federation/outbox-<rubric>.json   # анонсы версий по рубрике
/manifest.json                     # sha256-карта полного снапшота (export.py)
```

**Article карточки** (уже реализовано в `federate.py`):

| Поле | Значение |
|---|---|
| `id` | `<base>/federation/<card_id>` |
| `attributedTo` | actor хранителя (Person этого инстанса) |
| `updated` | `freshness.verified_at` |
| `content` / `source` | HTML / исходный Markdown |
| `zh:status`, `zh:version`, `zh:reviewDue` | датированная правда — машиночитаемо |

Поля `zh:*` — расширение протокола: без них реплика неотличима
от обычной статьи, с ними — проверяема на свежесть автоматически.

## 17.3. Подписка и синхронизация

Модель — «RSS с идентичностью», без двустороннего состояния:

```
инстанс B подписывается на outbox рубрики инстанса A
   → периодический poll outbox-<rubric>.json (рекомендуемый лаг: минуты—часы)
   → новая версия Article (сравнение zh:version)
   → B скачивает Article и помещает копию в свою зону «Реплики»
```

Правила синхронизации:

1. **Реплики не редактируются локально.** Правка реплики = правка
   через исходный инстанс (PR-модель там) или форк рубрики (§17.5).
2. **Конфликтов версий нет**, потому что у карточки один хозяин —
   инстанс происхождения. Версия хозяина всегда побеждает.
3. **Удаление не распространяется.** Если хозяин удалил карточку,
   реплики сохраняют последнюю версию с пометкой `ustarevaet`
   и ссылкой «удалена у источника» — история не стирается молча (У8).
4. **Свежесть реплики** — не хуже исходной: `zh:reviewDue` копируется;
   локальный RAG применяет к репликам те же фильтры статусов.

## 17.4. Журналы и реестр зеркал

- Журнал власти **не федеративен**: он описывает власть инстанса
  и экспортируется только как архив (в tar.gz снапшоте).
- Каждый инстанс публикует `_mirrors.yml` — список своих зеркал.
  Федерация образует **реестр выживания**: инстансы подписываются
  на манифесты друг друга и кэшируют чужие снапшоты. Смерть одного
  инстанса (dead man's switch) → соседи уже имеют его канон
  и могут поднять реплику-преемника.

## 17.5. Форк рубрики

Единственный способ «поссориться» конструктивно: сообщество инстанса B
забирает рубрику A целиком — копирует карточки в свой канон как
**форк** с обязательной атрибуацией (`forked_from: <инстанс A>`,
начальная версия = версии источника). С момента форка версии расходятся
независимо; RAG обоих инстансов честно показывает обе линии.
Исторический аналог — форк репозитория; юридически это возможно,
потому что канон — открытые данные.

## 17.6. Угрозы федерации (дополнение к документу 16)

| Угроза | Защита протокола |
|---|---|
| Инстанс-токсин: рассылает ложные карточки | Реплики помечены происхождением; подписка — opt-in; отписка мгновенна; рейтинг источников не передаётся |
| Подмена снапшота в транзите | sha256-манифест + подпись релиза (уровень 1) |
| Размывание власти («чужой садовник решил») | Журналы не федеративны; власть по построению локальна |
| Эхо-федерация: все копируют одно | Форк — право, а не сбой; метрика: доля собственных карточек инстанса |

## 17.7. Соответствие реализации (прототип)

| Пункт протокола | Реализация |
|---|---|
| Article + zh:* поля | `federate.py` → `dist/federation/*.json` |
| outbox рубрик | `dist/federation/outbox-*.json` |
| Манифест снапшота | `export.py` → `dist/export/manifest.json` |
| Реестр зеркал | `_mirrors.yml` |
| Dead man's switch | `deadman.py` + CI-алерт |
| Уровень 1 (живой сервер, подписи, двусторонние анонсы) | вне прототипа — форматы не изменятся |
