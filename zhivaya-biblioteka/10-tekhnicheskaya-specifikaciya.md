# 10. Техническая спецификация

Документ для разработчиков: форматы данных, API, события, ИИ-пайплайн.
Дополняет документ 02 (архитектуру) до уровня «можно начинать строить».

## 10.1. Хранение: карточка как файл

Каждая карточка — Markdown-файл с YAML-фронтматтером в Git-репозитории.
Структура репозитория:

```
canon/
  formy/
    agregatory/
      kn-2026-0417-moderaciya-v-reddit.md
      kn-2025-0098-golosovanie-kak-moderaciya.md
    nejroseti/
      ...
  _rubrics.yml          # дерево рубрик с хранителями
  _ledger.log           # журнал действий власти (append-only)
  _constitution/        # карточки устава
```

Полный формат файла карточки:

```markdown
---
id: kn-2026-0417
title: "Как устроена модерация в Reddit"
status: aktualno
version: 14
freshness:
  verified_at: 2026-03-14
  review_due: 2026-09-14
keeper: marina_k
coauthors: [anna_p, pavel_d]
rubrics: [formy/agregatory, moderaciya/modeli]
links:
  - {to: kn-2025-0098, type: utochnyaet}
  - {to: kn-2026-0331, type: protivorechit}
sources:
  - {url: "https://...", title: "Reddit Transparency Report 2025", checked: 2026-03-14}
aliases: [модерация реддит, реддит модерация]
---

# Как устроена модерация в Reddit

Тело карточки в обычном Markdown...

## История изменений
<!-- генерируется из git log, не редактируется вручную -->
```

**Почему файлы, а не БД:** экспорт = `git clone`; версии = `git log`;
форк = `git fork`; зеркало = `git push` на второй remote. База данных —
производная (индексы строятся из файлов), а не источник истины.

## 10.2. API (REST, черновик v1)

### Карточки
| Метод | Endpoint | Назначение | Права |
|---|---|---|---|
| GET | `/api/v1/cards/{id}` | Карточка + метаданные | все |
| GET | `/api/v1/cards/{id}/history` | Версии с диффами | все |
| GET | `/api/v1/cards/{id}?v=12` | Конкретная версия | все |
| POST | `/api/v1/cards/{id}/changes` | Предложить правку | участник+ |
| GET | `/api/v1/cards?status=trebuet-revizii` | Карточки на ревизию | все |

### Правки (changes)
| Метод | Endpoint | Назначение | Права |
|---|---|---|---|
| GET | `/api/v1/changes?rubric=X&state=open` | Очередь ревью рубрики | хранитель |
| POST | `/api/v1/changes/{id}/accept` | Принять (merge) | хранитель рубрики |
| POST | `/api/v1/changes/{id}/return` | Вернуть (body.comment обязателен — enforced) | хранитель |

### Вопросы, треды, журнал
| Метод | Endpoint | Назначение |
|---|---|---|
| POST | `/api/v1/questions` | Задать вопрос (ИИ-поиск запускается до публикации) |
| POST | `/api/v1/questions/{id}/distill` | Ответ → черновик карточки |
| GET | `/api/v1/ledger?actor=marina_k` | Журнал власти (публичный, read-only) |
| GET | `/api/v1/feed/{rubric}.rss` | RSS-лента рубрики |

## 10.3. Событийная шина (webhooks)

Все события системы — в открытой шине (на неё подписаны доставка, ИИ и зеркала):

```json
{
  "event": "change.accepted",
  "ts": "2026-07-19T12:40:00Z",
  "card": "kn-2026-0417",
  "version": 15,
  "author": "pavel_d",
  "reviewer": "marina_k",
  "diff_summary": "+2 абзаца о ценах на API",
  "ledger_id": "ldg-99213"
}
```

Типы событий: `card.created`, `change.proposed`, `change.accepted`,
`change.returned`, `card.status_changed`, `question.opened`,
`question.distilled`, `keeper.rotated`, `ledger.appended`.

## 10.4. ИИ-пайплайн (RAG, детально)

```
ИНДЕКСАЦИЯ (фон, при каждом change.accepted):
  карточка → чанки (по разделам, ~500 токенов, с якорями §)
           → эмбеддинги → векторный индекс
           → полнотекстовый индекс (BM25)
  метаданные чанка: card_id, version, status, verified_at, §-якорь

ОТВЕТ НА ВОПРОС (online):
  вопрос → гибридный поиск (вектор ∩ BM25), top-8 чанков
        → фильтр: исключить status in (ustarevaet, trebuet-revizii)
          ИЛИ включить с янтарной пометкой «устаревает»
        → генерация ответа СТРОГО с цитатами чанков
        → пост-проверка: каждое утверждение ответа привязано
          к card_id? Нет → ответ отклонён, ИИ отвечает
          «в каноне этого нет» + предлагает открыть вопрос
        → в футере: карточки-источники, их версии и даты проверки

ДИСТИЛЛЯЦИЯ ЧАТА (по кнопке «в канон»):
  тред → выделение решения/фактов → черновик карточки
      (структура: проблема → причина → решение → источники)
      → соавторы = участники треда → заявка хранителю

СКАНЕР УСТАРЕВАНИЯ (еженедельный cron):
  для каждой карточки: review_due прошёл? внешние ссылки живы (HEAD-запросы)?
  противоречия с новыми карточками (по графу links)?
  → заявки хранителям, никогда — автоправки
```

**Железное правило реализации:** у ИИ-пайплайна нет write-доступа к канону.
Только чтение + создание черновиков-заявок. Проверяется на уровне ACL.

## 10.5. Федерация и зеркала (формат обмена)

- **Зеркало** = git-remote, подписанный на шину событий: каждое
  `change.accepted` → pull. Лаг зеркала — минуты.
- **Федеративный анонс:** карточка сериализуется в ActivityPub-объект
  `Article` с `attributedTo` = хранитель, `updated` = версия. Другие
  инстансы Библиотеки могут подписываться на рубрики друг друга.
- **Dead man's switch:** если инстанс не подавал признаков жизни N дней,
  автоматический пайплайн публикует полный tarball канона + журнала
  на заранее настроенные публичные адреса (IPFS/HTTP-зеркала).

## 10.6. Нефункциональные требования

| Параметр | Цель |
|---|---|
| Чтение карточки | <200 мс (статика + CDN) |
| Поиск | <500 мс |
| Ответ ИИ-навигатора | <5 сек |
| Полный экспорт канона | <10 мин для 100k карточек |
| Потеря данных | 0 (git × 3 зеркала в разных юрисдикциях) |
| Доступность чтения | 99,9% (запись может деградировать, чтение — нет) |
