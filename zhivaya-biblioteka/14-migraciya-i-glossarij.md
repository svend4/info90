# 14. Playbook миграции сообществ + глоссарий

## Часть A. Миграция: как перевезти живое сообщество в Библиотеку

Исторический урок: сообщества не «появляются», а **переезжают**
(Reddit ← Digg, Mastodon ← Twitter). Миграция — штатная операция,
а не надежда на самопроизвольное зарождение.

### A.1. Предпосылки (когда миграция имеет смысл)

✔ Сообщество уже страдает: знания тонут в чате, форум умирает,
FAQ не обновлялся годами — есть **осязаемая боль**.
✔ Есть 3–5 уважаемых участников («ядро»), готовых стать хранителями.
✔ Есть архив, который жалко потерять (это и есть сид-контент).

✘ Сообществу «и так нормально» — миграция провалится, боли нет.
✘ Ядро против, а хочет администрация — власть не переезжает приказом.

### A.2. План на 90 дней

**Недели 1–2. Разведка.**
- Картирование: где живут знания (чат, форум, файлы, головы старожилов).
- Переговоры с ядром: предложение ролей хранителей, а не «юзеров».
- Выбор первой рубрики (узкой!): «не вся тема, а её самая болезненная часть».

**Недели 3–6. Перегон архива (ИИ + люди).**
- ИИ-дистилляция лучших тредов форума/чатов в 100–300 карточек-черновиков
  (уровень источника C — «опыт сообщества», честная пометка).
- Ядро ревьюит черновики: это и есть обучение хранителей на своём материале.
- Итог: рубрика, которая **уже отвечает** на топ-50 вопросов сообщества.

**Недели 7–10. Мягкий запуск.**
- В старом чате/форуме — бот-навигатор: на повторный вопрос отвечает
  ссылкой на карточку («это уже есть в Библиотеке, вот карточка»).
- Правило ядра: новые решения сначала в Библиотеку, потом анонс в чат.
- Публичная метрика миграции: «сколько вопросов недели закрыто карточками».

**Недели 11–13. Точка невозврата.**
- Старый форум — в режим «только чтение» (архив, не конкурент).
- Чат остаётся потоком — но с культурой «📌 в канон».
- Праздник миграции: первые выборы садовника новой рубрики.

### A.3. Типовые провалы миграции (и обход)

| Провал | Причина | Обход |
|---|---|---|
| «Перевезли архив — никто не ходит» | Архив ≠ живые ответы; нет бота-навигатора в старом месте | Миграция начинается с потока вопросов, не со склада |
| Ядро саботирует | Им предложили «аккаунты», а не власть и статус | Хранители из ядра с первого дня, их имена на карточках |
| Две площадки навсегда | Старое место не закрыли вовремя | «Только чтение» на 11-й неделе — неторопливо, но необратимо |
| ИИ-черновики низкого качества оттолкнули | Дали сообществу сырую дистилляцию | Люди-ядро ревьюят ВСЁ до показа сообществу |

## Часть B. Глоссарий проекта

| Термин | Значение |
|---|---|
| **Канон** | Совокупность актуальных карточек — «память» системы |
| **Карточка** | Версионируемая единица знания с хранителем и датой проверки |
| **Поток** | Чаты и лента событий — «жизнь», не хранилище |
| **Дистилляция** | Превращение треда/чата в черновик карточки (ИИ + человек) |
| **Кристаллизация** | Переход информации из потока в канон |
| **Хранитель** | Ответственный за рубрику: ревью, свежесть, вопросы |
| **Садовник** | Выборная роль: гигиена системы, ротации, конфликты карт |
| **Арбитр** | Разбирает апелляции и процедуры, не трогает контент |
| **Наставник** | Провожает новичка до 3-й принятой правки |
| **Свежесть** | Дата проверки карточки + дата обязательной ревизии |
| **Сиротская карточка** | Карточка без хранителя (красный сигнал) |
| **Журнал власти** | Публичный append-only лог всех действий ролей |
| **Песочница** | Тренировочная зона новичка, «где нельзя сломать» |
| **Режим осады** | Кризис-протокол при атаке ботов (двойное ревью, лимиты) |
| **Dead man's switch** | Автопубликация полного зеркала при гибели инстанса |
| **Уровень источника** | S/A/B/C/X — иерархия доверия к фактам (док. 11) |
| **RAG-only** | Правило ИИ: ответ только из цитат карточек |
| **«Вечный сентябрь»** | Исторический закон 1993 года: наплыв новичков размывает культуру быстрее, чем она успевает учить |
| **Конверсия чата в канон** | Главная метрика: сколько решений из потока кристаллизовано |
