# 02. Архитектура

## 2.1. Семь слоёв системы

```
┌─────────────────────────────────────────────────────┐
│ 7. ДОСТАВКА      подписки: RSS / email / канал      │
├─────────────────────────────────────────────────────┤
│ 6. НАВИГАЦИЯ     каталог · граф связей · поиск · ИИ │
├─────────────────────────────────────────────────────┤
│ 5. ВОПРОСЫ       Q&A, ответы кристаллизуются в канон│
├─────────────────────────────────────────────────────┤
│ 4. ДИСКУССИИ     треды, привязанные к объектам      │
├─────────────────────────────────────────────────────┤
│ 3. ПОТОК         чаты и лента (эфемерный слой)      │
├─────────────────────────────────────────────────────┤
│ 2. КАНОН         версионируемые карточки знаний     │
├─────────────────────────────────────────────────────┤
│ 1. ДОВЕРИЕ       роли · репутация · журнал власти   │
└─────────────────────────────────────────────────────┘
        Сквозь все слои: ХРАНИЛИЩЕ (Markdown + Git,
        открытый экспорт, федерация, зеркала)
```

### Слой 1. Доверие (фундамент)
Роли, репутация, публичный журнал действий власти. Не «фича», а инфраструктура:
любое действие в верхних слоях подписано ролью и попадает в журнал.
Подробно — в документе 04.

### Слой 2. Канон (память)
Ядро системы. Набор **карточек знаний** — версионируемых страниц в Markdown,
хранящихся в Git-подобном репозитории. Каждая карточка — как статья Википедии,
но с метаданными уровня базы данных (см. объектную модель ниже).

### Слой 3. Поток (жизнь)
Чаты по разделам + лента событий («в карточке X принята правка», «задан вопрос Y»).
Сообщения живут 90 дней в горячем виде, дальше — архив холодного хранения.
**Жёсткое правило:** кнопка «Решено» в чате не закрывает вопрос — она открывает
процедуру кристаллизации в карточку.

### Слой 4. Дискуссии (спор)
Треды, жёстко привязанные к объектам: к карточке (как Issues/PR-комментарии),
к вопросу, к правке. Нет «обсуждений в никуда» — каждый тред знает свой объект
и при разрешении предлагает итог в канон. Это лечит главную болезнь форумов:
обсуждение не отделено от предмета.

### Слой 5. Вопросы (дефицит знания)
Q&A-слой как «датчик дыр» в каноне. Вопрос — не пост, а **заявка на знание**.
Принятый ответ автоматически становится черновиком новой карточки или правки —
так Stack Overflow не окаменевает, а обновляется.

### Слой 6. Навигация (карта)
Четыре способа найти:
- **Каталог** — курируемое дерево рубрик (как Yahoo! Directory, но поддерживаемое
  хранителями + ИИ, поэтому масштабируется);
- **Граф связей** — гиперссылки между карточками с типами («уточняет», «противоречит», «пример»);
- **Полнотекстовый поиск** — по канону, тредам и вопросам раздельно;
- **ИИ-навигатор** — диалоговый поиск, отвечающий только ссылками на карточки.

### Слой 7. Доставка (почта)
Хронологические подписки: RSS, email-дайджест, канал-зеркало.
Пользователь подписывается на рубрику, карточку, человека или поисковый запрос.
Никакой алгоритмической подмешки — пришло ровно то, что заказано, в порядке времени.

## 2.2. Объектная модель

### Карточка знания (главный объект)

```yaml
id: kn-2026-0417
title: "Как устроена модерация в Reddit"
status: aktualno            # chernovik | aktualno | ustarevaet | trebuet-revizii
freshness:
  verified_at: 2026-03-14   # когда факты последний раз проверяли
  review_due: 2026-09-14    # когда обязательна ревизия
keeper: "@marina_k"         # хранитель — живой ответственный
rubrics: [formy/agregatory, moderaciya]
links:
  - {to: kn-2025-0098, type: "chast-ot"}
  - {to: kn-2026-0331, type: "protivorechit"}
body: |
  ...текст в Markdown...
history: git-log            # полная история версий с авторами и причинами
threads: [th-8812, th-9044] # привязанные обсуждения
questions: [q-3320, q-4105] # связанные вопросы Q&A
```

### Прочие объекты

| Объект | Аналог | Ключевое отличие |
|---|---|---|
| **Правка (Change)** | Pull Request | Правка любого текста: статьи, ответа, рубрики |
| **Тред (Thread)** | Тема форума | Привязан к объекту; имеет итог, который уходит в канон |
| **Вопрос (Question)** | Stack Overflow | Это заявка на пополнение канона, а не просто пост |
| **Сообщение (Msg)** | Чат | Имеет кнопку «в канон» и срок горячей жизни |
| **Рубрика (Rubric)** | Раздел каталога | Имеет хранителя и автосборку новых карточек |
| **Профиль (Profile)** | Соцсеть | Репутация считается по принятым правкам, не по постам |

## 2.3. Потоки информации

### Поток A. Кристаллизация (поток → канон)
```
Сообщение в чате "вот как мы решили проблему X"
   → ИИ замечает "похоже на решение" и предлагает дистилляцию
   → участник жмёт "в канон" (1 клик, не подвиг)
   → ИИ собирает черновик карточки из треда
   → хранитель области получает заявку-ревью
   → принятие → карточка в каноне, участники треда — соавторы
```

### Поток B. Обновление (канон → канон)
```
Читатель замечает устаревший факт
   → предлагает правку (интерфейс "как в Google Docs", под капотом — ветка git)
   → ИИ проверяет: не противоречит ли другим карточкам, добавляет источники
   → хранитель ревьюит (SLA: 72 часа, иначе эскалация к садовнику)
   → merge → версия +1, подписчики получают дифф в дайджесте
```

### Поток C. Дефицит (вопрос → канон)
```
Вопрос задан → ИИ ищет в каноне
   ├─ нашёл → ответ-ссылка + "карточка была полезна?" (сигнал качества)
   └─ не нашёл → вопрос публикуется, подписка хранителей области
        → принятый ответ → черновик карточки → поток B
```

### Поток D. Старение (фон)
```
ИИ еженедельно сканирует канон:
   → карточки с review_due < today → статус "trebuet-revizii", пинг хранителю
   → ссылки на внешние источники умерли → пометка link rot, поиск замены в архиве
   → две карточки противоречат → заявка на разрешение конфликта
   → хранитель неактивен 90 дней → ротация роли
```

## 2.4. Технологический минимум (если строить)

- **Хранилище:** Git-репозитории с Markdown (как у статических сайтов-документации).
- **Движок:** статический генератор + поисковый индекс (Meilisearch/Elasticsearch).
- **Обсуждения:** Discourse-подобный движок, связанный по ID объектов.
- **Доставка:** RSS-фиды по каждой рубрике + email-дайджесты.
- **ИИ:** RAG поверх канона (ответы только с цитатами карточек), черновики правок.
- **Федерация:** экспорт в git + ActivityPub-анонсы обновлений (опционально).

Принцип: **никакой магии в хранении.** Если завтра платформа исчезнет,
у сообщества остаётся папка с Markdown и полной историей — как и было у людей с Git.
